\section*{CI Pipeline 說明}

\texttt{.github/workflows/ci\_b12705041.yaml} 如下

\begin{lstlisting}[language=YAML]
name: CI b12705041

on:
    push:
        branches:
            - "**" # 所有 branch push 都觸發
    pull_request:  #  PR 也觸發，保護 main branch

permissions:
    contents: read # 讀取 repo 內容，actions/checkout 需要這個權限
    checks: write  # dorny/test-reporter 需要這個才能寫入 GitHub Checks 系統，才能讓測試通過/失敗的圖示出現在 Actions 頁面

jobs:
    ci:
        runs-on: ubuntu-latest # GitHub 幫你開一台全新的 Ubuntu 虛擬機來跑這些步驟。

        steps:
            - name: Checkout # 把你 repo 的程式碼下載進 VM。幾乎每個 workflow 的第一步都是這個，沒有 code 後面什麼都跑不了。
              uses: actions/checkout@v5

            - name: Setup Node.js
              uses: actions/setup-node@v5
              with:
                  node-version: "22"
                  cache: npm  #優化：把 node_modules 的下載結果快取起來。下次跑 CI 時如果 package-lock.json 沒變，就直接用快取

            - name: Install dependencies
              run: npm ci

            - name: TypeScript typecheck
              run: npm run typecheck #實際執行的是 tsc --noEmit，意思是「用 TypeScript 編譯器檢查型別，但不要輸出任何檔案」。

            - name: Prettier check #實際執行 prettier --check .，檢查所有程式碼的格式是否符合 .prettierrc 的規定（縮排、引號、分號等）
              run: npm run format:check #這個步驟只「檢查」，不會自動修正。格式不對就直接 fail。

            - name: Lint
              run: npm run lint

            - name: Run tests #--reporter=default：印在 log 裡，讓你在 Actions 頁面的 log 看到測試結果，--reporter=junit：輸出 vitest-junit.xml，這是業界標準的 XML 格式，給下一步的 test-reporter 用
              run: |
                  mkdir -p reports
                  npm test -- --reporter=default --reporter=junit --outputFile.junit=reports/vitest-junit.xml

            - name: Publish test results
              uses: dorny/test-reporter@v1
              if: always() # 不管測試成功或失敗都要執行這個步驟，這樣才能在 GitHub Actions 頁面看到測試結果
              with:
                  name: Vitest Test Results
                  path: reports/vitest-junit.xml
                  reporter: java-junit

\end{lstlisting}

CI Pipeline 的流程如下：
push 觸發
    ↓
開一台乾淨的 Ubuntu VM
    ↓
下載 code（Checkout）
    ↓
安裝 Node 22（Setup Node.js）
    ↓
安裝套件（npm ci）
    ↓
型別檢查（typecheck）→ 失敗就停
    ↓
格式檢查（prettier check）→ 失敗就停
    ↓
跑測試（vitest）→ 失敗會繼續跑下一步
    ↓
發布測試報告（dorny/test-reporter）← if: always() 確保一定跑
    ↓
VM 刪除

\texttt{[warn] .github/workflows/ci_b12705041.yaml}
\texttt{[warn] .github/workflows/01_hello.yaml}
\texttt{[warn] docker-compose.yml ...}
\texttt{Code style issues found in 7 files. → exit code 1}
\texttt{Prettier 發現這些檔案的格式不符合 .prettierrc 規定（縮排、引號等），pipeline 在這步就停了。}

\texttt{連鎖反應}

prettier check ❌ 失敗停止
    ↓
lint 沒跑
test 沒跑          ← reports/vitest-junit.xml 從來沒被產生
    ↓
dorny/test-reporter (if: always()) 雖然還是跑
    ↓
找不到 reports/vitest-junit.xml → "No test report files were found" ❌
所以「No test report」不是獨立的問題，是 Prettier 失敗的連鎖結果。

修復方式
讓 Prettier 自動幫你把所有檔案格式化：


npm run format
git add -A
git commit -m "style: format all files with prettier"
git push origin main
npm run format 執行 prettier --write .，自動修正所有格式問題。

推上去之後 CI 應該就能全部通過。

根本原因：
tsconfig.json 第 2 行："include": ["src/**/*.ts"] — 只包含 src/ 的檔案
ESLint 用 parserOptions.project: './tsconfig.json' 做型別感知分析
所以 test/app.test.ts 和 vitest.config.ts 不在 tsconfig 裡，ESLint 就報錯
最乾淨的修法：建一個 tsconfig.eslint.json 專門給 ESLint 用，包含所有要 lint 的檔案，不動原本的 tsconfig.json（那個是給 build 用的）。

為什麼這樣修？
tsconfig.json	給 npm run build 用，只需要 src/，不需要 test 檔案進入 production build
tsconfig.eslint.json	給 ESLint 用，extends 原本的設定，但額外包含 test/ 和 vitest.config.ts
這是 @typescript-eslint 官方推薦的標準做法，把「build 用的 tsconfig」和「lint 用的 tsconfig」分開。
